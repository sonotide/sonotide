# Основы Sonotide

## Что такое Sonotide

Sonotide — независимый C++20-фреймворк для Windows-аудио. Он даёт единый способ работать с устройствами, форматами, потоками и воспроизведением, не протаскивая в публичный API COM-объекты и детали WASAPI.

Фреймворк рассчитан на то, чтобы его можно было подключить к плееру, утилите записи, диагностическому инструменту или к собственному аудиосервису. При этом Sonotide не пытается заменить приложение целиком: он отвечает за аудиослой, а не за интерфейс, каталог треков, синхронизацию аккаунтов или бизнес-логику продукта.

## Границы

Внутри Sonotide находятся:

- перечисление render и capture устройств;
- выбор устройства по умолчанию или явного `device_id`;
- согласование аудиоформата;
- потоки вывода, захвата и loopback;
- высокоуровневая `playback_session` поверх потока вывода;
- встроенный 10-полосный эквалайзер для тракта воспроизведения;
- модель `result<T>` и типизированные ошибки;
- примеры, тесты и CMake package config.

Снаружи остаются:

- интерфейс пользователя;
- плейлисты, медиатека и пользовательские настройки;
- сетевые протоколы и аккаунтная логика;
- любые DSP-цепочки конкретного приложения, которые не входят во встроенный EQ;
- политика транспорта, если она нужна конкретному продукту поверх `playback_session`.

## Основные принципы

- Публичный API не знает о COM и Win32-деталях владения ресурсами.
- Объекты runtime и stream устроены как явные RAII-обёртки.
- Callback и жизненный цикл не скрываются за неявными фоновыми потоками.
- Поддерживаемый путь должен работать явно, а неподдерживаемый должен возвращать ошибку, а не притворяться успешным.
- Форматы, состояния и ошибки описаны типами, а не строками.
- Базовый сценарий для Windows — событийный shared-режим.

## Что уже входит в репозиторий

Sonotide уже поставляется как полноценный пакет:

- `runtime` для входа в библиотеку;
- фасады потоков для вывода, захвата и loopback;
- `playback_session` с загрузкой источника, декодированием и EQ;
- внутренний Windows backend;
- заголовки публичного API;
- examples и unit tests;
- документация по архитектуре, API и миграции.

## Минимальный сценарий

```cpp
auto runtime_result = sonotide::runtime::create();
if (!runtime_result) {
    return;
}

sonotide::runtime audio = std::move(runtime_result.value());
auto devices_result = audio.enumerate_devices(sonotide::device_direction::render);
if (!devices_result) {
    return;
}

for (const auto& device : devices_result.value()) {
    // Здесь можно вывести id, имя и состояние endpoint-а.
}
```

## Как читать документацию

- `docs/architecture.md` описывает слои, владение ресурсами и потоки исполнения.
- `docs/api.md` показывает, как пользоваться публичным API на практике.
- `docs/wasapi-mapping.md` связывает типы Sonotide с низкоуровневым WASAPI.
- `docs/migration-plan.md` нужен, если Sonotide внедряется в существующий проект поэтапно.

## Для чего этот фреймворк не подходит

Sonotide не стоит брать как контейнер для всей аудио-логики приложения. Если нужно сложное приложение с очередями, библиотекой, облаком, синхронизацией и графическим редактором, Sonotide должен остаться нижним слоем. Он не заменяет доменную модель продукта и не подменяет бизнес-логику.

## Что считать хорошим использованием

Хороший сценарий для Sonotide выглядит так:

- приложение выбирает устройство и формат через `runtime`;
- поток получает только свою часть работы через callback;
- `playback_session` управляет декодированием и транспортным состоянием;
- приложение хранит только свои пользовательские правила и метаданные.

Такой подход делает код предсказуемым и позволяет использовать Sonotide как библиотеку, а не как ещё один монолитный backend.
