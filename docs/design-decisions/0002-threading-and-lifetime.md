# ADR 0002: Потоки и жизненный цикл

## Контекст

Sonotide управляет потоками, которые взаимодействуют с WASAPI и внешним кодом через callback. Для такой схемы важно, чтобы было понятно, где выполняется код пользователя, кто владеет ресурсами и когда именно объект можно считать завершённым.

Если скрыть рабочий поток внутри фреймворка, то появятся неочевидные зависимости по времени жизни: вызовы могут продолжать выполняться после логического завершения объекта, а ошибки будут всплывать в неподходящий момент. Для аудио это особенно неприятно, потому что состояние устройства и `runtime` должно оставаться предсказуемым.

## Решение

Каждый активный поток владеет собственным рабочим потоком.

Основные правила жизненного цикла такие:

- `start()` запускает поток и ждёт завершения инициализации;
- callback-логика выполняется на этом же рабочем потоке;
- `stop()` останавливает работу и ждёт завершения потока;
- `reset()` сбрасывает согласованное состояние `runtime`, но не превращает объект в новый экземпляр;
- `close()` завершает объект окончательно и делает дальнейшее использование недопустимым.

Такой подход сохраняет явное владение ресурсами. Пользователь API понимает, когда поток ещё жив, когда он только подготовлен и когда он уже закрыт окончательно.

## Последствия

Плюс этого решения в том, что поведение потока становится предсказуемым и легко отлаживается. Нет скрытых фоновых задач, которые “живут где-то отдельно” от объекта.

Минус заключается в том, что ошибки синхронизации нельзя прятать за магией фреймворка. Если приложение вызывает методы в неправильном порядке, это должно быть видно сразу в состоянии потока или в результате операции. Sonotide сознательно выбирает явную модель вместо скрытой автоматизации.
