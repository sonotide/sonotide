# План миграции на Sonotide

Этот план написан для случая, когда в приложении уже есть рабочий backend на WASAPI, но он слишком тесно связан с транспортом, декодированием и логикой интерфейса. Цель миграции не в том, чтобы переписать всё одним куском, а в том, чтобы постепенно вынести переиспользуемый аудиослой в Sonotide и оставить в приложении только то, что действительно относится к продукту.

## Базовое правило

Мигрировать нужно по уровням, а не по принципу «вырезать старый файл и надеяться, что всё соберётся».

- если приложение уже умеет декодировать поток и ему нужен только аудиовыход, сначала переносится `runtime` и `render_stream`;
- если нужен цельный слой воспроизведения, можно подключать `playback_session` и уже потом решать, что оставить в приложении;
- захват и loopback лучше переводить отдельно, когда путь вывода уже стабилен;
- старый WASAPI backend удаляется только после того, как новый путь прошёл сборку, ручную проверку и базовые тесты.

## Шаг 1. Подключить Sonotide как зависимость

На этом этапе нужно просто сделать библиотеку доступной в сборке приложения и не менять поведение.

Что меняется:

- появляется зависимость на Sonotide CMake target;
- в коде приложения добавляются include-ы `sonotide/*.h`;
- прямой доступ к `IMMDevice`, `IAudioClient` и `IAudioRenderClient` остаётся только внутри временного адаптера.

Что проверить:

- приложение собирается без изменения поведения `runtime`;
- список аудиоустройств в интерфейсе или логах совпадает со старым вариантом;
- старый backend ещё можно включить обратно, если нужно сравнение.

## Шаг 2. Перенести перечисление устройств

Это самый безопасный первый перенос. Он не требует декодирования и не затрагивает DSP.

```cpp
auto runtime_result = sonotide::runtime::create();
if (!runtime_result) {
    return;
}

sonotide::runtime audio_runtime = std::move(runtime_result.value());
auto devices_result = audio_runtime.enumerate_devices(sonotide::device_direction::render);
if (!devices_result) {
    return;
}
```

После этого приложение должно перестать знать про COM-объекты устройств. В интерфейсе остаются только `device_info` и выбранный `device_id`.

Что проверить:

- render devices и capture devices отображаются одинаково с прежним backend;
- default device выбирается по той же роли;
- никакие app-facing сервисы больше не хранят `IMMDevice`.

## Шаг 3. Перевести путь вывода

Если приложение уже декодирует PCM само, ему нужен только адаптер к `render_callback`.

```cpp
class pcm_render_callback final : public sonotide::render_callback {
public:
    sonotide::result<void> on_render(
        sonotide::audio_buffer_view buffer,
        sonotide::stream_timestamp) override {
        // Здесь приложение берёт уже декодированные PCM-кадры
        // и копирует их в buffer.bytes.
        return sonotide::result<void>::success();
    }
};
```

Дальше поток открывается через `runtime::open_render_stream(...)`, и именно Sonotide управляет рабочим потоком, согласованием формата и жизненным циклом потока.

Если приложению нужен не только путь вывода, а полный тракт воспроизведения, на этом шаге можно перейти на `runtime::open_playback_session(...)`. Тогда декодирование, транспорт и EQ живут уже внутри верхнего слоя Sonotide, а приложение работает с `load`, `play`, `pause` и `seek_to`.

Что проверить:

- поток открывается на реальном устройстве;
- `render_stream::status()` показывает согласованный формат;
- старый код на `IAudioClient` больше не нужен в основной ветке вывода.

## Шаг 4. Перенести захват и loopback

Когда вывод уже работает через Sonotide, можно отдельно перевести входящие сценарии.

- обычный захват переводится на `runtime::open_capture_stream(...)`;
- loopback переводится на `runtime::open_loopback_stream(...)`;
- обработчики ошибок и stop/reset semantics должны стать одинаковыми для всех трёх типов потоков.

Что проверить:

- callback захвата получает данные без потери формата;
- loopback действительно слушает output device, а не микрофон;
- `stream_status.device_lost` корректно поднимается при invalidation устройства.

## Шаг 5. Перенести восстановление и состояние

После того как потоки работают через Sonotide, нужно убрать разрозненный код восстановления из приложения.

Что переносится в Sonotide-ориентированный слой:

- реакция на потерю устройства;
- повторное открытие потока;
- хранение согласованного формата и статистики `runtime`;
- выдача ошибок через `result<T>` и `error`.

Что остаётся в приложении:

- user-facing сообщение об ошибке;
- решение, нужно ли автоматически переподключаться;
- выбор новой target-установки, если текущая исчезла.

## Шаг 6. Удалить legacy WASAPI backend

Этот шаг делается только после того, как новый путь отработал достаточно долго, чтобы не было сомнений в паритете.

Удалять можно:

- прямое использование `IMMDeviceEnumerator`;
- прямое использование `IAudioClient`, `IAudioRenderClient`, `IAudioCaptureClient`;
- продуктовую обвязку вокруг `windows_streaming_audio_backend.cpp`;
- временные прослойки, которые только прокидывали вызовы в старый backend.

Оставить в приложении можно:

- декодирование;
- транспортную политику;
- интерфейс воспроизведения;
- persistence настроек;
- продуктовую логику поверх Sonotide.

## Рекомендуемая проверка после каждого шага

- `cmake --build --preset msvc-x64-debug`
- `ctest --preset msvc-x64-debug`
- `sonotide_list_devices.exe` для проверки перечисления устройств
- `sonotide_render_silence.exe` для проверки пути вывода
- `sonotide_playback_session.exe C:\Windows\Media\notify.wav` для проверки верхнего слоя воспроизведения, если он уже подключён

## Практический смысл миграции

Если всё сделать поэтапно, приложение не теряет работающий путь в середине перехода. Сначала Sonotide берёт на себя то, что одинаково для любого Windows-аудиоклиента, а потом уже можно решать, нужен ли ещё и верхний слой воспроизведения или достаточно одного слоя вывода и захвата.
